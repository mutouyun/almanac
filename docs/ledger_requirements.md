# Almanac Ledger 需求分析文档 v1.15

## 1. 项目背景
老板目前通过“iOS 快捷指令 + Webhook + 飞书多维表”实现移动端快速记账。为了实现数据的完全自主掌控和更灵活的报表分析，决定在 `almanac` 系统中自建记账模块，接管 Webhook 流量并沉淀至本地 SQLite 数据库。

## 2. 业务流程
1.  **数据采集**：移动端发起 HTTP POST 请求，包含 `date` (含时分), `type`, `amount` 三要素。
2.  **路由引擎**：系统基于用户定义的**正则表达式**，按**匹配层级深度与同层拖拽排序**，将原始输入匹配至某个分类节点（任意层级，深度优先）。
3.  **持久化**：数据写入本地 SQLite，并实现多用户账本隔离。
4.  **历史迁移**：支持通过 CSV 导入功能将历史数据迁入，并利用路由引擎自动对齐分类。
5.  **应用展示**：提供多层级汇总看板及明细查询。
6.  **数据导出**：支持将当前筛选视图的账目导出为 CSV（格式兼容导入，可再导回），与导入形成数据自主闭环。

## 3. 核心功能需求

### F1. 智能 Webhook 路由引擎
*   **输入协议**：接收 JSON 格式数据：
    *   `date`: 记账时间，需为**带时区偏移的 ISO 8601** 格式（如 `2026-07-05T14:30:00+08:00`）。移动端采集时携带本地系统时区，后端统一解析并归一化为东八区墙钟时间（精确到分）后存储。避免用户出国时手机时区切换导致账单时间乱序。
    *   `type`: 原始描述文本（如 "瑞幸咖啡"）。
    *   `amount`: 金额（**绝对值/无符号**，如 `"19.9"`）。收支方向不再由金额正负号承载，而由命中的分类决定（见下方「收支方向」）。**以字符串传递**，避免 JSON 浮点序列化的 IEEE 754 精度丢失；后端用 `json.Number` 或字符串解析交 decimal 处理，严禁 float64 接入参。为兼容旧客户端，若入参仍带符号，后端取其绝对值存储、符号忽略。
*   **判定逻辑**：
    *   **收支方向**：**由命中分类的方向决定**（分类本身带 `direction`：收入类 / 支出类），与金额正负号完全脱钩。金额在库中存**无符号绝对值**，方向为分类的派生属性。若一笔账目匹配不到任何分类（待分类状态），则**暂无方向**，由用户在明细里手动指定其属于支出分类还是收入分类，从而确定方向。
    *   **正则归类**：系统遍历该用户**所有分类节点（任意层级，不限叶子）**。配了正则的分类：普通模式存裸关键词（包含匹配），高级模式由用户手写正则（如 `^...$` 精确匹配）。**未配正则的分类：默认按分类名做精确全等匹配**（等价于自动生成 `^分类名$` 正则，分类名经元字符转义、严格区分大小写；匹配前对原始描述去除前后空白）。因此所有分类节点都参与自动路由。
    *   **匹配优先级**：由于方向不再由金额符号预先锁定，路由引擎在**收入类与支出类合并后的全部分类**中统一匹配；按**层级深优先**（子节点优先于父节点）；同一父节点下的兄弟由用户拖拽排序决定先后（每一层均可拖拽）；跨子树同深度平局按确定性规则兑底。命中即止，命中分类的方向即为该账目的收支方向。
    *   **匹配冲突解决**：若命中多个正则，按上述**层级深度 + 同层 `sort_order` 拖拽序**确定顺序，第一个命中的节点为最终归类点（无独立 weight 字段）。
    *   **溯源填充**：匹配成功后，系统自动关联该节点向上追溯的所有父层级分类（最高支持 5 级）。账目可归到任意层级节点，不强制落到叶子。
    *   **兜底机制**：未匹配到任何分类时，记录存为“待分类”状态（`category_id` 为空，既无分类也无方向）；待分类账目**不计入收入也不计入支出汇总**，在看板单独列“待分类”提醒用户处理；用户在明细里手动指定分类后即获得方向并计入汇总。

### F2. 递归分类体系（多用户隔离）
*   **多用户账本**：系统支持多用户体系。每个用户拥有一个独立账本，其分类树、路由规则与账目数据完全物理/逻辑隔离。
*   **分类树域界（Scope）= 用户级共享**：分类体系为**用户级**资源，不随账本分裂。即使将来升级单用户多账本（如“个人账”“家庭账”），所有账本仍共用同一棵分类树。因此修改分类名称或方向将波及该用户所有账本的历史归类，交互层需慎重。
*   **五级动态树**：支持最多 5 层的树状嵌套分类结构，由用户完全自定义。
*   **节点约束**：
    *   **方向继承**：子节点必须强制继承父节点的收支方向（收入类或支出类），且方向创建后不可修改。
    *   **全员记账与路由**：任意层级节点均可被账目归属，也均可配置正则参与自动路由（不再限定叶子）。
*   **管理后台**：支持 UI 化的树形结构维护、正则编辑及同层排序（sort_order 拖拽）调整。

### F3. 数据持久化 (SQLite)
*   **架构设计**：采用本地 SQLite 存储，利用递归查询 (Recursive CTE) 实现高效的多级报表聚合统计（如：统计一级大类时自动累加其下所有子孙节点的金额）。

### F4. 历史数据迁移 (CSV 导入)
*   **导入逻辑**：支持解析飞书导出的 CSV 格式。导入时利用路由引擎尝试自动对齐。无法对齐的记录进入“待确认列表”供用户手动校对。

### F5. 账号与 Webhook Token 管理
*   **Token 生成**：用户账号创建时自动生成一个密码学安全的随机 `webhook_token`（`crypto/rand` 生成 ≥256 bit 熵，带可读前缀如 `alm_`）并存入 `users.webhook_token`。用户开箱即有，无需手动创建。
*   **Token 获取与展示**：用户登录后在「设置/账号安全页」查看自己的 token（默认打码，可显示/一键复制），并获得快捷指令配置示例（URL + Header + Body 模板）。
*   **Token 重置**：支持「重新生成」（带二次确认），旧 token 立即失效，用于泄露补救或定期轮换。
*   **安全约束**：token 的查看与重置端点**必须走登录会话鉴权**（不得用 token 自身换 token，防泄露后无限续命）；token 在 Header 明文传输，生产环境必须 HTTPS。
*   **修改密码**：用户可在设置页修改登录密码（校验当前密码 + 新密码强度）。

### F6. 账本数据导出 (CSV)
*   **业务价值**：与 F4「CSV 导入」形成数据自主闭环——用户既能把历史账单迁入，也能把 almanac 里的账本随时导出留存/备份/二次分析，呼应「数据完全自主掌控」的项目初衷，不被系统绑定。
*   **导出范围 = 当前视图**：导出的是**账目明细页当前筛选条件下的全部账目**（方向 / 分类 / 关键词 / 时间区间 / 金额区间），而非仅当前分页可见的一页；无筛选时即导出该用户全部账目。与列表页共用同一套筛选条件，保证「看到的即导出的」。
*   **格式自洽（可再导入）**：导出为 UTF-8（带 BOM，防 Excel 打开乱码）CSV，列集是 F4 导入格式（`账单日 / 类别 / 子类别 / 金额`）的**超集**，额外附 `方向 / 来源` 便于人工阅读。因此一份导出文件可直接再走 F4 导入回系统（导入侧忽略未知列、按表头名取所需列）。
*   **金额与方向**：金额导出为**无符号 magnitude**（与导入侧一致，方向由分类派生而非金额符号承载）；`方向` 列为「收入 / 支出 / 未分类」人类可读标签。
*   **权限与隔离**：走登录会话鉴权，仅导出当前用户自己账本的数据（`user_id` 隔离）。

## 4. 非功能性需求
*   **安全性**：Webhook 接口需配置独立的 `webhook_token` 进行鉴权。token 为长效 API Key 模式（非会话），快捷指令无状态调用不触发登录；token 获取/重置见 F5。
*   **时区管理**：严格锁定东八区（Asia/Shanghai），确保包含时分的记账记录时间准确无偏差。
*   **扩展性**：模型设计需预留账本 ID 字段，以便未来平滑升级到单用户多账本模式。
