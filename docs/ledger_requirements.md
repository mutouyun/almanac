# Almanac Ledger 需求分析文档 v1.11

## 1. 项目背景
老板目前通过“iOS 快捷指令 + Webhook + 飞书多维表”实现移动端快速记账。为了实现数据的完全自主掌控和更灵活的报表分析，决定在 `almanac` 系统中自建记账模块，接管 Webhook 流量并沉淀至本地 SQLite 数据库。

## 2. 业务流程
1.  **数据采集**：移动端发起 HTTP POST 请求，包含 `date` (含时分), `type`, `amount` 三要素。
2.  **路由引擎**：系统基于用户定义的**正则表达式**，按**匹配层级深度与同层拖拽排序**，将原始输入匹配至某个分类节点（任意层级，深度优先）。
3.  **持久化**：数据写入本地 SQLite，并实现多用户账本隔离。
4.  **历史迁移**：支持通过 CSV 导入功能将历史数据迁入，并利用路由引擎自动对齐分类。
5.  **应用展示**：提供多层级汇总看板及明细查询。

## 3. 核心功能需求

### F1. 智能 Webhook 路由引擎
*   **输入协议**：接收 JSON 格式数据：
    *   `date`: 记账时间，需为**带时区偏移的 ISO 8601** 格式（如 `2026-07-05T14:30:00+08:00`）。移动端采集时携带本地系统时区，后端统一解析并归一化为东八区墙钟时间（精确到分）后存储。避免用户出国时手机时区切换导致账单时间乱序。
    *   `type`: 原始描述文本（如 "瑞幸咖啡"）。
    *   `amount`: 金额（支出为负，收入为正）。**以字符串传递**（如 `"-19.9"`），避免 JSON 浮点序列化的 IEEE 754 精度丢失；后端用 `json.Number` 或字符串解析交 decimal 处理，严禁 float64 接入参。
*   **判定逻辑**：
    *   **收支方向**：严格基于 `amount` 正负号判定（非类型映射）。
    *   **正则归类**：系统遍历该用户**所有配了正则的分类节点（任意层级，不限叶子）**。普通模式存裸关键词（包含匹配），高级模式由用户手写正则（如 `^...$` 精确匹配）。
    *   **匹配优先级**：按**层级深优先**（子节点优先于父节点）；同一父节点下的兵弟由用户拖拽排序决定先后（每一层均可拖拽）；跨子树同深度平局按确定性规则兑底。命中即止。
    *   **匹配冲突解决**：若命中多个正则，按上述**层级深度 + 同层 `sort_order` 拖拽序**确定顺序，第一个命中的节点为最终归类点（无独立 weight 字段）。
    *   **溯源填充**：匹配成功后，系统自动关联该节点向上追溯的所有父层级分类（最高支持 5 级）。账目可归到任意层级节点，不强制落到叶子。
    *   **兜底机制**：未匹配到任何正则时，记录存为“待分类”状态，并在后台引导用户补全规则。

### F2. 递归分类体系（多用户隔离）
*   **多用户账本**：系统支持多用户体系。每个用户拥有一个独立账本，其分类树、路由规则与账目数据完全物理/逻辑隔离。
*   **分类树域界（Scope）= 用户级共享**：分类体系为**用户级**资源，不随账本分裂。即使将来升级单用户多账本（如“个人账”“家庭账”），所有账本仍共用同一棵分类树。因此修改分类名称或方向将波及该用户所有账本的历史归类，交互层需慎重。
*   **五级动态树**：支持最多 5 层的树状嵌套分类结构，由用户完全自定义。
*   **节点约束**：
    *   **方向继承**：子节点必须强制继承父节点的收支方向（收入类或支出类），且方向创建后不可修改。
    *   **全员记账与路由**：任意层级节点均可被账目归属，也均可配置正则参与自动路由（不再限定叶子）。
*   **管理后台**：支持 UI 化的树形结构维护、正则编辑及同层排序（sort_order 拖拽）调整。

### F3. 数据持久化 (SQLite)
*   **架构设计**：采用本地 SQLite 存储，利用递归查询 (Recursive CTE) 实现高效的多级报表聚合统计（如：统计一级大类时自动累加其下所有子孙节点的金额）。

### F4. 历史数据迁移 (CSV 导入)
*   **导入逻辑**：支持解析飞书导出的 CSV 格式。导入时利用路由引擎尝试自动对齐。无法对齐的记录进入“待确认列表”供用户手动校对。

## 4. 非功能性需求
*   **安全性**：Webhook 接口需配置独立的 `webhook_token` 进行鉴权。
*   **时区管理**：严格锁定东八区（Asia/Shanghai），确保包含时分的记账记录时间准确无偏差。
*   **扩展性**：模型设计需预留账本 ID 字段，以便未来平滑升级到单用户多账本模式。
